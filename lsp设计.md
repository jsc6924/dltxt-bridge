# 概要
我要设计一个vscode lsp服务器。最终目标是用户打开一个写有日语的文本文件，lsp给vscode推送：
 - 对于目前这个文本文件中的每一个句子，lsp推送当前项目中与之相似的句子列表（相似度大于某个阈值），以及这些句子所在的文件路径和行号。

# 设计
为了实现以上功能，我需要
## 能打开多种常见编码（如UTF-8、Shift_JIS、UTF16, GBK等）的文本文件，并将其转换成统一编码格式（UTF16）进行处理。
1. 使用uchardet识别文本文件的编码格式。
2. 使用iconv将文本文件转换成UTF16编码格式。

磁盘扫描到的工作区文件才需要做编码探测和转码。
LSP 的 didOpen 和 didChange 收到的是 VS Code 已经解码后的文本，不该再走 uchardet 和 iconv。

## TextDocument类
打开的文件会被封装成一个TextDocument类，包含以下属性：
- lines: 文件的文本内容（UTF16编码，将每行分割成一个字符串数组）
- encoding: 文件的原始编码格式
- filePath: 文件的绝对路径
- fileName: 文件名
然后支持以下方法：
- getLine(lineNumber): 获取指定行的文本内容
- getContent(): 获取整个文件的文本内容
- getEncoding(): 获取文件的原始编码格式
- getFilePath(): 获取文件的绝对路径
- getFileName(): 获取文件名
- edit(Range, newText): 编辑文件内容（如插入、删除、替换文本等，可能涉及到多行。但如果是单行操作，必须只修改该行内容）

TextDocument只是LSP在内存中的副本，不直接修改磁盘文件。

# 索引设计
1. 只需支持utf16编码。文件的单位是行（所以比较小），每个被索引的行有一个ID（唯一标识符），以及文件路径和行号。索引内部需要把文件名map到数字ID保存，以节省内存。
2. 索引需要支持增量更新。（把ID对应的行内容替换成新的内容）
3. 索引使用2-gram+3-gram的方式分词。
4. 索引需要支持检索操作（给定一个句子，返回一组与他相似的句子）。相似度通过TF-IDF向量化句子，并计算余弦相似度来衡量。向量必须以稀疏的形式储存（只存出现的词），以节省内存。
5. 索引只在内存中存在。

# 文本parser
我需要处理的文本具有以下这种格式：
```
○00027145T○昔から、<d0047>恋愛リアリティ番組</d>が好きだった。
●00027145T●我从小就喜欢看<d0047>真实恋爱节目</d>。

○00027146T○男女数人がルームシェアしたり、<br>１人の異性を皆で取り合ったり。
●00027146T●比如男女数人一起同住，或者几个人追求同一位异性。

○00027148N○華淡
●00027148N●華淡

○00027147T○（アタシが応援する子、大抵ロクな目に合わんけど）
●00027147T●（虽然我声援的女孩子，基本最后都没什么好结局）
```
为了提取句子，我需要一个parser。我在vscode插件端已经有一个这样的parser。我需要你在lsp服务器端重写这个parser。
为了方便你重写，我已经把整个插件的代码全都复制进了项目里（见dltxtSrc），你可以参考。当然，parser也要放在一个独立的文件里。对于getRegex函数，你需要向vscode端发送一个请求，让插件端返回正则表达式。插件端目前还没有实现这个rpc api，所以你可以直接修改这里复制的插件的源码，我会帮你把修改复制到插件端。


## 相似句子的计算
首先在bridge扫描整个项目中的文本文件，以句子（一行就是一句）为单位建立倒排索引，每个句子记录其所在的文件路径和行号。文件需要经过parser，提取出句子内容（text），去掉标签等装饰性的文本。只需要索引日语部分（jgrps）即可，cgrps直接跳过。搜索的时候也需要提取出text搜索，并且只搜索日语部分（jgrps）。换句话说，当前文本切换时，vscode会传来一个request，告诉当前文本的uri，lsp需要读取这个文本文件的内容，提取出所有jgrps的text部分，对于每一个text搜索索引中相似的句子，然后把结果返回给vscode（结果需要包含相似句的text，以及所在的文件路径和行号）。

索引细节：
1. 分词：使用2-gram+3-gram的方式
2. 相似度计算：使用TF-IDF向量化句子，并计算余弦相似度。（向量的储存必须是稀疏的，只存出现的词）
3. 阈值：设定一个相似度阈值，只有相似度大于该阈值的top K句子才进入下一步
4. 对于通过上一步的句子，与原句用编辑距离计算相似度，然后根据相似度排序，返回前N个最相似的句子列表。

当用户编辑文本文件时，lsp需要及时更新索引中的句子内容，以保证相似度计算的准确性。
磁盘上的文件如果发生变化，不需要实时监听。索引可以允许滞后性。

## LSP服务器设计
建立索引必须使用线程池完成，不能占用主线程。
每个窗口的bridge都建立一套自己的索引，不共享。
计算相似度时也必须使用线程池完成，不能占用主线程。
主线程必须在内存里维持vscode中打开的文件的副本（TextDocument实例），监听vscode的文件打开、编辑、保存等事件，及时更新TextDocument实例和索引的内容。

主线程在得知用户切换到一个文本文件时（onDidChangeActiveTextEditor事件），会把一个任务提交给线程池，线程池会计算该文件中每个句子与索引中句子的相似度，并将结果推送给vscode。
不过，计算的结果可以缓存，如果用户在短时间内切换回同一个文件，可以直接使用缓存的结果，而不需要重新计算。
