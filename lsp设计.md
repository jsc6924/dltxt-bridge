# 概要
我要设计一个vscode lsp服务器。最终目标是用户打开一个写有日语的文本文件，lsp给vscode推送：
 - 对于目前这个文本文件中的每一个句子，lsp推送当前项目中与之相似的句子列表（相似度大于某个阈值），以及这些句子所在的文件路径和行号。

# 设计
为了实现以上功能，我需要
## 能打开多种常见编码（如UTF-8、Shift_JIS、UTF16, GBK等）的文本文件，并将其转换成统一编码格式（UTF16）进行处理。
1. 使用uchardet识别文本文件的编码格式。
2. 使用iconv将文本文件转换成UTF16编码格式。

磁盘扫描到的工作区文件才需要做编码探测和转码。
LSP 的 didOpen 和 didChange 收到的是 VS Code 已经解码后的文本，不该再走 uchardet 和 iconv。

## TextDocument类
打开的文件会被封装成一个TextDocument类，包含以下属性：
- lines: 文件的文本内容（UTF16编码，将每行分割成一个字符串数组）
- encoding: 文件的原始编码格式
- filePath: 文件的绝对路径
- fileName: 文件名
然后支持以下方法：
- getLine(lineNumber): 获取指定行的文本内容
- getContent(): 获取整个文件的文本内容
- getEncoding(): 获取文件的原始编码格式
- getFilePath(): 获取文件的绝对路径
- getFileName(): 获取文件名
- edit(Range, newText): 编辑文件内容（如插入、删除、替换文本等，可能涉及到多行。但如果是单行操作，必须只修改该行内容）

TextDocument只是LSP在内存中的副本，不直接修改磁盘文件。

## 相似句子的计算
首先在bridge扫描整个项目中的文本文件，以句子（一行就是一句）为单位建立倒排索引，每个句子记录其所在的文件路径和行号。
索引细节：
1. 分词：使用2-gram+3-gram的方式
2. 相似度计算：使用TF-IDF向量化句子，并计算余弦相似度。（向量的储存必须是稀疏的，只存出现的词）
3. 阈值：设定一个相似度阈值，只有相似度大于该阈值的top K句子才进入下一步
4. 对于通过上一步的句子，与原句用编辑距离计算相似度，然后根据相似度排序，返回前N个最相似的句子列表。

当用户编辑文本文件时，lsp需要及时更新索引中的句子内容，以保证相似度计算的准确性。
磁盘上的文件如果发生变化，不需要实时监听。索引可以允许滞后性。

## LSP服务器设计
建立索引必须使用线程池完成，不能占用主线程。
每个窗口的bridge都建立一套自己的索引，不共享。
计算相似度时也必须使用线程池完成，不能占用主线程。
主线程必须在内存里维持vscode中打开的文件的副本（TextDocument实例），监听vscode的文件打开、编辑、保存等事件，及时更新TextDocument实例和索引的内容。

主线程在得知用户切换到一个文本文件时（onDidChangeActiveTextEditor事件），会把一个任务提交给线程池，线程池会计算该文件中每个句子与索引中句子的相似度，并将结果推送给vscode。
不过，计算的结果可以缓存，如果用户在短时间内切换回同一个文件，可以直接使用缓存的结果，而不需要重新计算。
